# Nginx配置验证说明

## 为什么nginx配置中使用 `prajna-tts-server:8000`

### Docker网络通信原理

当两个容器在同一个Docker网络（如 `web`）中时，它们可以通过容器名称互相访问，Docker会自动进行DNS解析。

```
容器A (nginx)  →  Docker网络(web)  →  容器B (prajna-tts-server)
   ↓                                           ↓
访问: prajna-tts-server:8000          监听: 0.0.0.0:8000
```

### 端口说明

#### 1. 容器内部端口（8000）

这是TTS服务在容器内部监听的端口：

```dockerfile
# Dockerfile中
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

#### 2. 宿主机端口映射（-p 8000:8000）

```bash
docker run -p 8000:8000 ...
         ↑    ↑
    宿主机端口 容器端口
```

这个映射的作用：
- 将容器的8000端口映射到宿主机的8000端口
- 允许从宿主机直接访问：`http://localhost:8000`
- 方便测试和调试

#### 3. Nginx配置中的端口

```nginx
upstream tts_backend {
    server prajna-tts-server:8000;
}
```

这里的 `:8000` 是指：
- 容器内部的8000端口
- 不是宿主机的8000端口
- 通过Docker网络直接访问

### 访问方式对比

#### 方式1: 从宿主机访问（需要端口映射）

```bash
# 需要: -p 8000:8000
curl http://localhost:8000/health
```

#### 方式2: 从Docker网络内访问（nginx使用）

```bash
# 不需要端口映射，只需要在同一网络
docker exec sub-nginx curl http://prajna-tts-server:8000/health
```


### 常见问题

#### Q1: 为什么不用 `localhost:8000`？

A: 在Docker容器中，`localhost` 指的是容器自己，不是宿主机或其他容器。

```bash
# 错误 ❌
server localhost:8000;  # nginx容器的localhost，不是TTS容器

# 正确 ✅
server prajna-tts-server:8000;  # 通过容器名称访问
```

#### Q2: 为什么不用宿主机IP？

A: 使用容器名称更好：
- 容器名称在Docker网络中自动解析
- 不依赖宿主机IP（IP可能变化）
- 更安全（不需要暴露到宿主机）
- 性能更好（直接通过Docker网络）

```bash
# 不推荐 ⚠️
server 192.168.1.100:8000;  # 宿主机IP，需要端口映射

# 推荐 ✅
server prajna-tts-server:8000;  # 容器名称，直接通信
```

#### Q3: 端口映射 `-p 8000:8000` 是必需的吗？

A: 对于nginx访问TTS服务，不是必需的。但建议保留，原因：
- 方便直接测试TTS服务
- 方便调试和排查问题
- 不影响nginx的访问

```bash
# 最小配置（nginx能访问，但不能从宿主机直接访问）
docker run --network web prajna-tts-server

# 推荐配置（nginx能访问，也能从宿主机直接访问）
docker run --network web -p 8000:8000 prajna-tts-server
```

### 完整的网络流程

#### 外部用户访问流程

```
用户浏览器
    ↓ HTTPS请求
    ↓ https://tts.zhdnpx.cn/tts/section/mp3?section_id=ch1-1
    ↓
宿主机:443 (nginx监听)
    ↓
nginx容器 (sub-nginx)
    ↓ 反向代理
    ↓ http://prajna-tts-server:8000/tts/section/mp3?section_id=ch1-1
    ↓ (通过Docker网络web)
    ↓
TTS容器 (prajna-tts-server)
    ↓ 处理请求
    ↓ 生成音频
    ↓
返回mp3文件
    ↓ (通过Docker网络web)
    ↓
nginx容器
    ↓ 返回给用户
    ↓
用户浏览器播放音频
```

#### 直接测试流程（通过宿主机端口）

```
开发者
    ↓ HTTP请求
    ↓ http://localhost:8000/health
    ↓
宿主机:8000 (端口映射)
    ↓
TTS容器:8000
    ↓ 处理请求
    ↓
返回 {"ok": true}
```

### 验证清单

部署完成后，按顺序验证：

- [ ] 1. TTS容器运行正常
  ```bash
  docker ps | grep prajna-tts-server
  ```

- [ ] 2. TTS容器在web网络中
  ```bash
  docker network inspect web | grep prajna-tts-server
  ```

- [ ] 3. TTS服务健康检查（容器内部）
  ```bash
  docker exec prajna-tts-server curl http://localhost:8000/health
  ```

- [ ] 4. TTS服务健康检查（宿主机）
  ```bash
  curl http://localhost:8000/health
  ```

- [ ] 5. nginx能访问TTS服务（Docker网络）
  ```bash
  docker exec sub-nginx curl http://prajna-tts-server:8000/health
  ```

- [ ] 6. nginx配置语法正确
  ```bash
  docker exec sub-nginx nginx -t
  ```

- [ ] 7. nginx反向代理正常（HTTP）
  ```bash
  curl -I http://tts.zhdnpx.cn/health
  # 应该返回301重定向到HTTPS
  ```

- [ ] 8. nginx反向代理正常（HTTPS）
  ```bash
  curl https://tts.zhdnpx.cn/health
  # 应该返回 {"ok": true}
  ```

- [ ] 9. 测试音频生成
  ```bash
  curl "https://tts.zhdnpx.cn/tts/section/mp3?section_id=ch1-1&voice=male" -o test.mp3
  # 应该生成test.mp3文件
  ```

- [ ] 10. 测试CORS
  ```bash
  curl -I -H "Origin: https://your-app.com" https://tts.zhdnpx.cn/health
  # 应该看到 Access-Control-Allow-Origin: *
  ```

### 故障排查命令

```bash
# 1. 查看TTS容器日志
docker logs prajna-tts-server

# 2. 查看nginx日志
docker logs sub-nginx
docker exec sub-nginx tail -f /var/log/nginx/tts.zhdnpx.cn.error.log

# 3. 进入TTS容器调试
docker exec -it prajna-tts-server /bin/sh

# 4. 进入nginx容器调试
docker exec -it sub-nginx /bin/sh

# 5. 测试DNS解析
docker exec sub-nginx nslookup prajna-tts-server

# 6. 测试网络连通性
docker exec sub-nginx ping prajna-tts-server

# 7. 查看端口监听
docker exec prajna-tts-server netstat -tlnp

# 8. 重启服务
docker restart prajna-tts-server
docker restart sub-nginx
```
