# Nginx配置修复说明

## 问题描述

在执行 `nginx -s reload` 时遇到以下错误：

```
[warn] the "listen ... http2" directive is deprecated
[emerg] "add_header" directive is not allowed here in /etc/nginx/conf.d/tts.zhdnpx.cn.conf:87
```

## 问题分析

### 问题1: http2指令已弃用（警告）

**旧语法**（nginx < 1.25.1）：
```nginx
listen 443 ssl http2;
```

**新语法**（nginx >= 1.25.1）：
```nginx
listen 443 ssl;
http2 on;
```

### 问题2: add_header不能在if块中使用（错误）

**错误写法**：
```nginx
if ($request_method = 'OPTIONS') {
    add_header 'Access-Control-Allow-Origin' '*';  # ❌ 不允许
    return 204;
}
```

**原因**：
- nginx的 `if` 指令是rewrite模块的一部分
- `add_header` 是headers模块的指令
- 两者不能混用

## 解决方案

### 方案1: 简化OPTIONS处理（已采用）

直接在if块中返回204，CORS头部在server级别统一设置：

```nginx
server {
    # 在server级别设置CORS头部（对所有location生效）
    add_header 'Access-Control-Allow-Origin' '*' always;
    add_header 'Access-Control-Allow-Methods' 'GET, POST, HEAD, OPTIONS' always;
    add_header 'Access-Control-Allow-Headers' 'Origin, X-Requested-With, Content-Type, Accept, Range, Authorization' always;
    
    location /tts/ {
        # 简单处理OPTIONS请求
        if ($request_method = 'OPTIONS') {
            return 204;  # ✅ 只返回状态码，不设置头部
        }
        
        proxy_pass http://tts_backend;
    }
}
```

### 方案2: 使用map（备选）

如果需要更复杂的逻辑，可以使用map：

```nginx
map $request_method $cors_method {
    OPTIONS 204;
    default "";
}

server {
    location /tts/ {
        if ($cors_method) {
            return $cors_method;
        }
        proxy_pass http://tts_backend;
    }
}
```

### 方案3: 使用named location（备选）

```nginx
server {
    location /tts/ {
        error_page 418 = @options;
        if ($request_method = 'OPTIONS') {
            return 418;
        }
        proxy_pass http://tts_backend;
    }
    
    location @options {
        add_header 'Access-Control-Allow-Origin' '*' always;
        return 204;
    }
}
```

## 修复后的配置

### 主要变更

1. **HTTP/2配置**：
```nginx
# 旧版本
listen 443 ssl http2;

# 新版本
listen 443 ssl;
http2 on;
```

2. **CORS处理**：
```nginx
# 在server级别统一设置CORS头部
add_header 'Access-Control-Allow-Origin' '*' always;
add_header 'Access-Control-Allow-Methods' 'GET, POST, HEAD, OPTIONS' always;
add_header 'Access-Control-Allow-Headers' 'Origin, X-Requested-With, Content-Type, Accept, Range, Authorization' always;

# 在location中简单处理OPTIONS
location /tts/ {
    if ($request_method = 'OPTIONS') {
        return 204;
    }
    proxy_pass http://tts_backend;
}
```

3. **移除限制请求方法的if块**：
```nginx
# 删除了这段代码（会导致问题）
# if ($request_method !~ ^(GET|HEAD|POST|OPTIONS)$) {
#     return 405;
# }
```

## 兼容性说明

### Nginx版本要求

- **http2 on 指令**：需要 nginx >= 1.25.1
- **旧语法 (listen ... http2)**：nginx < 1.25.1

### 检查nginx版本

```bash
docker exec sub-nginx nginx -v
```

### 如果nginx版本较旧

如果你的nginx版本 < 1.25.1，使用旧语法：

```nginx
listen 443 ssl http2;
listen [::]:443 ssl http2;
# 不需要 http2 on;
```

## 其他配置文件

如果你有其他配置文件（如 `dbeaver.zhdnpx.cn.conf`）也有同样的警告，需要同样修复：

```bash
# 查找所有使用旧语法的配置文件
docker exec sub-nginx grep -r "listen.*http2" /etc/nginx/conf.d/

# 批量替换（在宿主机上）
cd /opt/nginx/conf.d/
sed -i 's/listen \(.*\) ssl http2;/listen \1 ssl;\n    http2 on;/g' *.conf
```

## 常见问题

### Q1: 为什么CORS头部要加 always 参数？

A: `always` 参数确保即使在错误响应（4xx, 5xx）中也添加CORS头部。

```nginx
add_header 'Access-Control-Allow-Origin' '*' always;
```

### Q2: 为什么不在每个location中单独设置CORS？

A: 在server级别设置CORS头部，所有location都会继承，避免重复配置。

### Q3: if块中还能使用哪些指令？

A: if块中只能使用rewrite模块的指令：
- `return`
- `rewrite`
- `set`
- `break`

不能使用：
- `add_header`
- `proxy_pass`（除非是唯一指令）
- 其他模块的指令

### Q4: 如何处理更复杂的CORS需求？

A: 使用map指令：

```nginx
map $http_origin $cors_origin {
    default "";
    "~^https?://(.*\.)?example\.com$" $http_origin;
}

server {
    add_header 'Access-Control-Allow-Origin' $cors_origin always;
}
```
